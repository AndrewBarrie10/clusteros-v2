# ClusterOS — Canonical Stack Definitions
**Version:** 1.0  
**Status:** Approved  
**Source:** Derived from Stage 3 (Stall Canon v2.1) and Stage 4 (Stack Canon v1.0) pipeline prompts  
**Embedded in:** clusteros-v2 (this repo) and clusteros-v3 (pipeline repo)

---

## Principles

Stacks are named descriptively, not evaluatively. Names follow the pattern: **[Mechanism] [Configuration]** — what the system is doing, not what is wrong with it.

Each stack entry includes canonical ID, name, component stalls, absorbed signals, reinforcing logic, leverage entry point, and downstream rules.

## Pair Configurations (20)

### STK-01 · Coordination–Narrative Configuration
**Stalls:** S2 + S7  
**Absorbed:** Friction, Uncertainty  
**Logic:** Coordination requires narrative to justify continued alignment; narrative requires coordination to appear credible and current.  
**Leverage entry point:** The reporting layer — changing what counts as evidence of progress may perturb both stalls simultaneously.  
**Confidence:** High

### STK-02 · Coordination–Intermediary Configuration
**Stalls:** S2 + S5  
**Absorbed:** Complexity, Friction  
**Logic:** Coordination creates demand for intermediary infrastructure; intermediaries make coordination manageable; each sustains the other.  
**Leverage entry point:** The coupling layer — one direct actor-to-actor connection that bypasses intermediaries demonstrates an alternative pathway.  
**Confidence:** High

### STK-03 · Validation–Tolerance Configuration
**Stalls:** S1 + S3  
**Absorbed:** Demand, Failure  
**Logic:** Re-proving keeps the option space open; tolerating non-performers reduces pressure to make commitments that would collapse options.  
**Leverage entry point:** The closure layer — one visible programme closure with named accountability may perturb option-keeping behaviour.  
**Confidence:** Medium

### STK-04 · Visibility–Activity Configuration
**Stalls:** S7 + S8  
**Absorbed:** Pressure, Uncertainty  
**Logic:** Activity generates narrative material; narrative legitimises further activity scaling; the system appears productive externally while conversion remains unmeasured.  
**Leverage entry point:** The measurement layer — one conversion metric (not an activity metric) creates visibility into throughput absence.  
**Confidence:** High

### STK-05 · Extraction–Incumbent Configuration
**Stalls:** S4 + S6  
**Absorbed:** Success, Disruption  
**Logic:** Incumbent institutions capture value that new actors generate; extraction reinforces incumbent centrality; disruption becomes less likely.  
**Leverage entry point:** The structural layer — changing anchor procurement or spin-out policy changes the constraint without requiring incumbents to change behaviour.  
**Confidence:** Medium

### STK-06 · Permission–Extraction Configuration
**Stalls:** S4 + S9  
**Absorbed:** Opportunity, Success  
**Logic:** Waiting for external permission delays autonomous action; while waiting, value continues to exit; exit reduces capacity for the autonomous action that would break the wait.  
**Leverage entry point:** The autonomy layer — one unilateral action that does not require permission creates a pathway others can follow.  
**Confidence:** Medium

### STK-07 · Coordination–Permission Configuration

**Confidence:** High

### STK-08 · Validation–Coordination Configuration

## Triple Configurations (10)

Rarer but diagnostically significant. These represent qualitatively different structural stability.

### STK-15 · Governance Capture Configuration
**Stalls:** S2 + S7 + S8  
**Absorbed:** Friction, Uncertainty, Pressure  
**Logic:** Coordination generates reports; reports justify programme activity; programme activity generates coordination requirements; all three signals absorbed simultaneously making the system self-referential.  
**Leverage entry point:** The external pressure layer — an outside actor who refuses coordination and demands direct behavioural evidence may create a gap in the self-referential loop.  
**Confidence:** High

### STK-16 · Permission–Validation Configuration
**Stalls:** S1 + S2 + S9  
**Absorbed:** Demand, Friction, Opportunity  
**Logic:** Re-proving requires coordination to appear credible; coordination requires permission to proceed; waiting extends the re-proving cycle; all three signals absorbed by the validation-permission loop.  
**Leverage entry point:** The unilateral action layer — one actor who validates by doing rather than by coordinating or seeking permission may break all three simultaneously.  
**Confidence:** Medium

### STK-17 · Extraction–Intermediary Configuration
**Stalls:** S4 + S5 + S6  
**Absorbed:** Complexity, Disruption, Success  
**Logic:** Incumbents extract value via intermediary relationships; intermediaries protect incumbent access; incumbent stability suppresses the disruption that would reduce extraction.  
**Leverage entry point:** The visibility layer — making extraction visible (talent flows, capital flows, procurement destinations) may create conditions for structural change without requiring incumbents to act differently.  
**Confidence:** Low

**Leverage entry point:** The permission architecture layer — demonstrating that one class of decision does not require this architecture may reduce its perceived necessity.  
**Confidence:** Medium

## Downstream Rules

### Self-Assessment
- Self-assessment identifies stalls (max 5); stacks are derived automatically from stall co-occurrence
- User sees canonical stack name and reinforcing logic — not a generated description
- Top 3 stacks shown, ranked by specificity (triples over pairs) then confidence
- Pairs that are strict subsets of a displayed triple are suppressed
- Health score reflects stall count only — not stack severity or leverage difficulty

### Leverage
- Every stack has a canonical leverage entry point
- Pipeline-generated leverage hypotheses must map to a canonical entry point
- AI commentary may reference the leverage principle but must frame it as a hypothesis, not a prescription

### Comparators
- Matching uses canonical stack IDs, not stall names alone
- Strong comparator: shares the same canonical stack(s)
- Partial comparator: shares 2+ component stalls but not the full stack
- Sector and regime are secondary filters, not primary

### AI Commentary
- Prompts must include canonical stack name and reinforcing logic
- Must not invent leverage beyond the canonical entry point
- Must use absorbed signals framing
- Must not rank stacks or declare severity
